# DaemonicCabal.jl ‚Äî Zig learning journal (for the Windows port)

## The goal
Learn Zig by porting this project (currently Linux/macOS/BSD) to Windows. Approach: tutor-led, small steps, OS internals woven in. User is Julia-fluent and polyglot; new to Zig. Developing on macOS; can cross-compile to Windows but can't run/test Windows binaries locally. Fast-paced, don't overproduce, ask trick questions, correct mistakes.

## The project
Daemon for Julia: keeps warm Julia workers alive so scripts run many times but compile once. Three components (README `Architecture`, line 199):
- **Client** (`juliaclient`, Zig, `client/`) ‚Äî users invoke; proxies stdio to a worker.
- **Conductor** (Zig daemon, `conductor/`) ‚Äî manages the worker pool, routes clients.
- **Worker** (Julia, `worker/`) ‚Äî the Julia process serving a project.

Zig parts: `client/` and `conductor/`. Each has `platform/` (OS abstraction) and `eloop/` (event loop: `kqueue.zig`, `linux.zig`, `posix_signals.zig`). `build.jl` already has commented-out Windows cross-compile targets (line 143-144). README lists supported platforms at line 44.

## Lessons covered

### L1 ‚Äî comptime platform selection (`client/platform/main.zig`)
- `builtin.os.tag` is a comptime enum; `if` on it is a comptime expression, so dead branches are eliminated and only the live `@import` is analyzed.
- `@compileError(msg)` halts compilation *of whatever references it* ‚Äî in a dead branch it's never seen. `@panic` would also compile (only fires if executed at runtime).
- The pattern: pick `impl` (linux/bsd/windows) at comptime, then `pub const SIG = impl.SIG;` etc. re-exports a uniform symbol set. Zero-cost abstraction.
- Exercise run: `zig run hello.zig` (native, runs) vs `zig build-exe -target x86_64-windows-gnu hello.zig` (cross-compile, no run). Zig is a cross-compiler out of the box.

### L2 ‚Äî the two-tier platform pattern (`main.zig` + `posix.zig`)
- Bucket 1: raw per-OS primitives (`SIG`, `getpid`, `write`, `kill`, `rawSocket`, `rawConnect`, `rawClose`, `defaultRuntimeDir`, `isLoopback`, std-handle getters) ‚Äî aliased straight from `impl`.
- Bucket 2: shared higher-level logic (`socketWrite`, `close`, `waitpidNonBlocking`, `getTerminalSize`, `isatty`, `setRecvTimeout`, `setRawMode`, `registerSignalHandlers`) ‚Äî built on bucket 1, lives in `posix.zig`.
- `main.zig:49`: `const shared = if (os != .windows) @import("posix.zig") else impl;` ‚Äî on Windows, `windows.zig` must provide *both* buckets itself (no POSIX to share).
- Inline `if` shims at lines 31-33, 69-72 paper over signature differences on Windows (`isLoopback`, `setRawMode`, `setWorkerRawMode`).
- **Key insight**: `posix.zig:10` has no `.windows` branch but compiles fine on Windows because it's only `@import`ed from inside the `os != .windows` arm of `main.zig:49`. Comptime branching prunes whole files from the import graph ‚Äî `posix.zig` can use POSIX-only types ungated.

### L3 ‚Äî the POSIX socket lifecycle (`client/client.zig:472` `notifyInterruptRaw`)
- Context: signal-handler-safe interrupt notify; can't touch io_uring state, so raw syscalls only.
- Canonical client pattern: `socket() ‚Üí connect() ‚Üí write()/read() ‚Üí close()`.
- **socket(family, type)** returns an `fd_t`. `AF.UNIX` = filesystem path, `AF.INET` = IPv4, `SOCK.STREAM` = reliable ordered stream. On POSIX, socket fds share the same integer namespace as files/pipes/stdio (0/1/2) ‚Äî *the same `write()` works on sockets*. On Windows: sockets are separate `SOCKET` handles, need `WSAStartup`, use `send`/`recv`/`closesocket`. This is why `socketWrite`/`socketRead`/`close` are routed through `shared`, not aliased from `impl`.
- **connect(fd, *sockaddr, len)** ‚Äî the C polymorphism hack: `connect` only knows generic `sockaddr`; you `@ptrCast` the concrete `sockaddr_un`/`sockaddr_in`/`sockaddr_in6` and the kernel reads `family` to re-interpret. `@ptrCast` is `reinterpret_cast`, no checks.
- **Network byte order**: wire is big-endian; `addr.port = std.mem.nativeToBig(u16, port)` swaps the port. `addr.addr = ip` is *not* swapped ‚Äî `parseIPv4` (line 510) builds `parts: [4]u8` in octet order and `@bitCast` preserves the byte layout in memory, which is already network order. `@bitCast` is memory reinterpretation, not numeric conversion ‚Äî works regardless of host endianness (the little-endian framing is incidental).
- **defer** queues a call for scope exit (success or error); `defer platform.rawClose(fd)` guarantees no fd leak even on early return. Critical for long-running daemons.
- **What a socket physically is**: a kernel object (struct in kernel memory) with a send buffer, a receive buffer, protocol state (TCP state machine / almost nothing for AF_UNIX), and bindings (addr+port or fs path). The `fd` is just a per-process index pointing at this object. `write` copies user‚Üísend buffer; `read` copies recv buffer‚Üíuser. For AF_UNIX one endpoint's send buffer feeds the other's receive buffer in-kernel ‚Äî basically a bidirectional pipe. Windows historically split socket handles from file handles ‚Äî root of the porting pain.

### L4 ‚Äî server socket lifecycle + event loops (`conductor/protocol.zig`, `conductor/eloop/kqueue.zig`)
- Server pattern extends L3: `socket ‚Üí bind ‚Üí listen ‚Üí accept ‚Üí read/write ‚Üí close`. `bind` attaches an address; `listen` flips the socket to passive mode (stops being readable/writable, becomes a connection acceptor). Zig's `Io.net.UnixAddress.listen` / `Io.net.IpAddress.listen` (`protocol.zig:275-285`) wrap socket+bind+listen.
- **Backlog / accept queue**: `listen(fd, backlog=128)` sets a kernel queue of *completed-but-not-yet-accepted* connections. A client's `connect()` can finish the TCP handshake *before* the server calls `accept()`; the connection sits in the queue. If full, new connects block/drop. This is why slow servers refuse connections ‚Äî queue full, not network broken.
- **`accept` clones the fd**: `accept(server_fd)` returns a *brand-new* fd for *one* connection; `server_fd` keeps listening. A server is one immortal listening fd + a stream of per-connection fds that come and go. `kqueue.zig:221` does `c.accept(...)`, then `defer c.close(client_fd)`. The `peer` sockaddr is the *client's* address, filled by the kernel.
- **Why an event loop**: naive `accept ‚Üí read ‚Üí handle` blocks on slow clients; one slow client freezes the daemon. Event loop = multiplexing: "tell me when *any* fd is ready, I'll handle just that one." `kqueue.zig:150`: `kevent` blocks until ‚â•1 registered event ready, returns a batch, loop processes each, loops back.
- **kqueue model**: kernel object (`c.kqueue()`) that you register `Kevent` interests on. Fields: `ident` (what to watch ‚Äî fd or synthetic id), `filter` (READ/TIMER/WRITE/SIGNAL‚Ä¶), `flags` (ADD/DELETE/ONESHOT/CLEAR), `udata` (opaque `usize` copied verbatim into the fired event ‚Äî *this is how you know which event fired*). Trick at lines 38-42: fixed sources get sentinels (`UDATA_ACCEPT=0`), worker events use `@intFromPtr(w)` ‚Äî the pointer stuffed into the usize, recovered with `@ptrFromInt`. Zero alloc, O(1) lookup.
- **Level-triggered (default)**: fires *while the condition holds*. 5 simultaneous connects ‚Üí server fd stays readable ‚Üí `kevent` returns one event per call across 5 iterations (NOT 5 events in one batch ‚Äî kqueue returns at most one event per (ident,filter) pair). Never miss a connection.
- **One-shot (`EV.ONESHOT`)**: fires once, auto-deletes. Used for ping read+timeout (`kqueue.zig:298-317`) ‚Äî each ping is a distinct cycle, auto-cleanup.
- **Edge-triggered (`EV.CLEAR`)**: fires only on the *transition* to readable; you must drain everything in one go (loop reads until EAGAIN) or never hear about it again. Not used here; high-performance/high-complexity.
- **Architectural tension exposed**: `handleAccept` is non-blocking for accept (kqueue says readable = connection waiting), but `handleConnectionFd` ‚Üí `readExact` is a **blocking read** on the client socket. A stalled client wedges the whole loop. Works in practice because clients send magic+request immediately (local, low latency) and worker sockets get `setRecvTimeout`. Proper fix: register `client_fd` with kqueue after accept, read only when readable.
- **Blocking vs non-blocking sockets** (key misconception fix): blocking socket (default) `read()` with no data ‚Üí **blocks the thread** (does NOT error). Non-blocking (`O_NONBLOCK` via `fcntl`) `read()` with no data ‚Üí returns `EAGAIN`/`EWOULDBLOCK` immediately. The minimal one-syscall fix to `handleAccept` would be `fcntl(fd, F_SETFL, O_NONBLOCK)`; the full fix is re-registering `client_fd` with kqueue.
- **Windows divergence ‚Äî IOCP**: kqueue/epoll are *readiness* models ("tell me when this fd is *ready* to read"). IOCP is a *completion* model ("I'm starting a read; tell me when it's *done*"). With IOCP you issue the read *before* data is ready; the kernel completes it later. The inverted semantics are why one event-loop abstraction can't cleanly paper over both ‚Äî the single biggest Windows porting challenge. Zig's `std.Io` (`protocol.zig:275` uses `Io.net.*.listen`) is the stdlib's attempt at a unified interface.
- **ONESHOT vs CLEAR for the ping pair**: the same-batch race (`if (!w.ping_pending) return;` guard) is handled regardless of trigger mode. But ONESHOT auto-deletes on fire (self-cleaning per cycle); CLEAR only changes *when* it fires, doesn't auto-delete. With CLEAR the TIMER would become periodic (re-firing forever after a successful pong unless explicitly DELETEd), and the READ would stay registered across pings, tangling state. ONESHOT is load-bearing here.

### L5 ‚Äî process spawning: fork+exec vs CreateProcess (`conductor/worker.zig:207` `spawnImpl`)
- Two spawn paths in `spawnImpl`: normal (`std.process.spawn`, line 321) and sandboxed (Linux-only raw `execve` after `fork`+`unshare`+bind-mounts, line 294).
- **Unix `fork()` + `execve()`**: `fork` clones the calling process (new PID, COW copy of address space, fd table duplicated). `execve(path, argv, envp)` replaces the *entire address space* with a fresh program from `path`; PID stays the same; fd table survives (modulo close-on-exec). "Duplicate me, then in the duplicate swap in a new program." The **window between fork and exec** is the *only* place to prepare the child's environment (cwd, fd redirection, namespaces) before the new program starts ‚Äî exploited by the sandbox path via `posix_spawn_file_actions` or raw fork/exec glue.
- **`.pgid = 0`** at line 323 puts the worker in a new process group (Unix only; `null` on Windows). So Ctrl-C at the terminal ‚Üí SIGINT to the *conductor's* group only, not the workers. Conductor routes interrupts via the protocol, not terminal signals. `.pgid = null` on Windows ‚Äî Windows has no process groups in this sense.
- **Sandbox path** (`worker.zig:254-298`): Linux only. `fork` + `unshare(CLONE_NEWUSER | CLONE_NEWNS | ...)` + bind-mounts + `execve`. The fork-exec window is used to set up a whole new mount namespace (isolated filesystem view). **No Windows analog** ‚Äî Windows sandboxing is Job Objects + restricted tokens / WSL / containers, totally different mechanism.
- **The connect-back stdio pattern**: (1) *Before* spawning, conductor creates a Unix-domain listening socket at `wsetup.sock` (line 236). (2) Spawns Julia with `--eval 'DaemonWorker.runworker("wsetup.sock", ...)'` ‚Äî the worker's first action is to connect back. (3) `setup.server.accept(io)` (line 326) accepts the callback. (4) Later in `runClient`: conductor tells worker to create **four more listening sockets** (stdin/stdout/stderr/signals), sends those paths to the client, client connects to all four, worker accepts them (`main.zig:1666-1709`).
  - Why connect-back instead of classic `pipe()`+`fork()`+`dup2()`+`exec()` fd inheritance: no fd-inheritance complexity; worker is a normal Julia process that picks its own socket paths (per-client, not per-worker); naturally sandbox-friendly (only the per-worker subdirectory needs bind-mounting).
- **Windows divergences (three pain points, increasing severity)**:
  1. **No `AF_UNIX` historically** ‚Äî Windows 10 1803+ added it but with caveats. Classic Windows equivalent is **named pipes** (`\\.\pipe\<name>`): `CreateNamedPipe` + `ConnectNamedPipe` per instance, client `CreateFile`. No `bind/listen/accept` model ‚Äî different semantics, the listen/accept abstraction from L4 doesn't carry over.
  2. **No `fork()`** ‚Äî Windows has only `CreateProcess` (one syscall: new process, fresh address space, no inherited fd table in the Unix sense). Handles inherit only if explicitly marked inheritable, passed via `STARTUPINFOW`. **No window between fork and exec** ‚Äî anything to set in the child's environment (cwd, env, stdio) must be in `CreateProcess` args up front. The sandbox path has no Windows equivalent at all.
  3. **No process groups / no `kill(-pgid, sig)`** ‚Äî Windows uses **Job Objects** (kernel object grouping processes; `TerminateJobObject` kills all) or console control events (`GenerateConsoleCtrlEvent`, targets a process group attached to a console ‚Äî different concept). The `kill(pid, SIG.INT)` path in `handleNotification` needs a Windows-specific mechanism.
- **Why the blocking `accept` at line 326 is *less* dangerous than the stalled-client read from L4**: the worker is a process we *just spawned* whose first action we dictated ‚Äî trusted, local, known choreography. Threat model: "trusted local process I birthed" vs "untrusted remote client." The failure window (spawn ‚Üí exec ‚Üí Julia startup ‚Üí connect) is small, and most failures are fast (bad executable ‚Üí immediate exit). Not arbitrary network input.
- **The remaining exposure**: if the worker crashes *between* spawn and `runworker`'s connect-back, the blocking `accept` wedges the conductor. Worse: the signal pipe is watched by kqueue, but we're not *in* the kqueue loop ‚Äî we're in a blocking syscall ‚Äî so even SIGCHLD-to-pipe conversion can't save us; the pipe-write happens but nobody reads it. `setRecvTimeout` at line 329 is applied *after* accept, to the data socket ‚Äî doesn't guard the accept. Missing mitigation: timeout on the setup listener, or non-blocking accept integrated with kqueue. Tolerated because Julia-startup-then-crash is rare and fast; recoverable via `--restart` or SIGTERM.

### L6 ‚Äî Unix domain sockets & the runtime dir (`conductor/protocol.zig:173`, `conductor/config.zig:57`, `conductor/platform/{linux,bsd}.zig:defaultRuntimeDir`, `conductor/main.zig:316`)
- **What `AF_UNIX` physically is**: a socket whose "address" is a *filesystem path* (not host:port). The kernel creates a special inode at that path (type `SOCK` in `ls -l`, shown as `s`). Two processes connect to the same path; the kernel wires their send/receive buffers together in-kernel. No network layer, no TCP, no port, no host resolution. A bidirectional byte stream addressed by a filename.
- **`sockaddr_un`** has a fixed-size `sun_path[108]` (Linux) / `[104]` (BSD) ‚Äî the load-bearing constraint. `client.zig:483` and `protocol.zig:181` both length-check before `@memcpy` into `addr.path`. Overflow would either truncate (silent wrong behavior) or panic deep in std's bind. The check surfaces it as a clear `error.PathTooLong`.
- **`randomSocketPath` (`protocol.zig:173`)** has *two* separate too-long checks: (1) `bufPrint` catch ‚Äî guards the caller's `path_buf` which is `[std.fs.max_path_bytes]u8` (~4096 Linux, ~1024 macOS), fires only on absurd paths; (2) `if (path.len >= sun_path_len)` ‚Äî guards the kernel's 108-byte `sun_path`, the binding constraint. Two limits, two buffers, two checks ‚Äî defensive layering.
- **Runtime dir requirements**: short path (fits `sun_path[108]`); reachable by both conductor and client (separate processes ‚Äî macOS's sandbox-private `/var/folders` fails this, see `bsd.zig:240-243`); secure (`0700` perms so other users can't connect); cleaned on logout/reboot (stale sockets cause `EADDRINUSE`).
- **Per-OS defaults** (README line 171-172): Linux `/run/user/$UID/julia-daemon` (XDG_RUNTIME_DIR, per-user tmpfs from systemd, 0700, auto-cleaned); macOS `~/Library/Application Support/julia-daemon` (no XDG_RUNTIME_DIR, and it's often sandbox-private anyway); BSD `/tmp/julia-daemon-$UID` (no standard, /tmp + UID suffix, less secure). `config.zig:57-60` honors `JULIA_DAEMON_RUNTIME` override first.
- **Stale socket file problem**: `AF_UNIX` listening socket creates a filesystem inode at `bind` time that *persists after close*. `bind` to existing path ‚Üí `EADDRINUSE`. Protocol: unlink-before-bind, unlink-after-close. `main.zig:307` and `worker.zig:238` use `defer Io.Dir.deleteFileAbsolute(...)` for cleanup-on-error. Hard crash (SIGKILL/power) skips defers ‚Üí stale files. `createServer`/`createListener` must tolerate existing paths.
- **`cleanupRuntimeDir` (`main.zig:316-329`)**: blanket-wipes the runtime dir (files + recursive subdir deletion via `deleteTree`). Called automatically at `main.zig:1971`, between `createDirPath` and `conductor.run()` ‚Äî every startup gets a clean slate. Necessary because stale sockets would block `bind`. **Blanket wipe assumes single-user-single-conductor**; a second conductor sharing the dir would have its sockets nuked.
- **Hard-crash failure mode (SIGKILL with 3 clients connected)**: kernel auto-closes all conductor fds ‚Üí kernel socket objects freed instantly (NOT a kernel-memory leak). What persists: filesystem `SOCK` inodes at the socket paths (dead, `connect` ‚Üí `ECONNREFUSED`). Cleanup wipes: `conductor.sock`, `conductor.pid`, every `wsetup.sock`, all `sandbox-{id}/` subdirs (recursively), all 12 client stdio sockets. What cleanup does NOT handle: **orphaned Julia worker processes** ‚Äî they're separate processes (no `PR_SET_PPR`), keep running, hold warm state, try to ping dead conductor, fail, eventually hit their own TTL exit and burn RAM until then. A "kill stale workers from previous run" step (via `conductor.pid`) would plug it; project doesn't do this.
- **Windows replacement: named pipes** (`\\.\pipe\<name>`): no `bind/listen/accept` ‚Äî `CreateNamedPipe` creates *one instance* per concurrent client, `ConnectNamedPipe` accepts on it, client `CreateFile` connects. To accept N clients: pre-pipe a pool or spawn a thread looping Create+Connect. No filesystem inode (kernel object, auto-cleaned when all handles close ‚Üí no stale-file problem, no `cleanupRuntimeDir` needed). Names up to 256 chars (no `sun_path[108]` limit). Security via DACL (Windows-specific), not `0700` perms. `defaultRuntimeDir` becomes mostly irrelevant for the pipe name itself, still needed for PID file / sandbox subdirs.
- **Windows 10 1803+ added real `AF_UNIX`** ‚Äî same socket/bind/listen/accept API. Possible porting shortcut: keep `AF_UNIX`, handle Windows-specific bits (no fork, different signals). Caveats: filesystem permissions on Unix sockets differ, still need `WSAStartup`. Project must decide: `AF_UNIX` (modern Windows only, simpler port) vs named pipes (broader compat, bigger rewrite of listen/accept layer).

### L7 ‚Äî signals & the self-pipe trick (`conductor/eloop/posix_signals.zig`, `client/platform/posix.zig:100-167`)
- **What a signal physically is**: the kernel *injects a function call* into your process at an unpredictable moment. Trigger: `kill(pid, sig)`, terminal driver (Ctrl-C ‚Üí SIGINT), kernel (segfault ‚Üí SIGSEGV, child died ‚Üí SIGCHLD). At the next syscall boundary (or between instructions), the kernel modifies the user-space stack/registers so control jumps to your registered handler instead of resuming. On return, `sigreturn` restores saved context. The handler runs *on top of whatever the thread was doing* ‚Äî this is the entire problem.
- **Async-signal-safety**: the handler must not touch state the interrupted thread might be holding locks for. POSIX safe list: `_exit`, `write`, `read`, `close`, `signal`, `sigaction`, `fork`‚Ä¶ **NOT safe**: `malloc`, `printf`, `fopen`, most stdio, anything taking a libc lock, anything touching global state. `std.debug.print` is NOT safe (can deadlock if interrupted thread was itself printing). This is why signal handlers in this project are tiny.
- **The self-pipe trick** (`posix_signals.zig`): do almost nothing in the handler ‚Äî write one byte to a pipe, return. The event loop watches the other end, wakes up, reads the byte, does the *real* handling in normal context where anything is safe to call. End-to-end: (1) `installSignalHandlers` creates `signal_pipe` via `Io.Threaded.pipe2(.{ .NONBLOCK = true })` (line 38); (2) `sigaction` installs `handleShutdown` for SIGTERM/SIGINT (line 44-45); (3) kqueue registers `signal_pipe[0]` readable (`kqueue.zig:123`); (4) signal arrives ‚Üí `handleShutdown` writes `'S'` to `signal_pipe[1]` ‚Üí returns; (5) kqueue wakes ‚Üí `UDATA_SIGNAL` ‚Üí `handleSignal` reads byte ‚Üí `gracefulShutdown()`. Signal converted from async interruption to synchronous event-loop event. One of the most beautiful Unix idioms.
- **`rawWrite` bypasses the platform abstraction** (`posix_signals.zig:20-26`): one raw `write` syscall, no `EINTR` retry, no `std.debug.print` on error (the platform `write` in `bsd.zig:26-40` has both ‚Äî unsafe from a handler). If write fails (pipe full, `.NONBLOCK` ‚Üí `EAGAIN`), signal silently dropped. Acceptable: SIGTERM is re-sent by OS/user.
- **`pipe2` with `O_NONBLOCK`** (`posix_signals.zig:38`): a blocking write in a handler would be catastrophic ‚Äî handler blocks forever if loop is busy draining, loop can't drain because handler sits on top. Non-blocking + drop-on-full is the safe choice.
- **Client-side handler ‚Äî different design** (`client/platform/posix.zig:100-167`): `signalAction` (line 125) does mode-selected routing: `.INT` ‚Üí if `worker_raw` write `\x03` to stdin (Julia's REPL line editor gets Ctrl-C as a byte), else `notifyInterrupt()` (conductor `kill`s worker with SIGINT ‚Üí `InterruptException` in running task). Same signal, two routing paths depending on `setWorkerRawMode` state. `.TERM`/`.HUP` ‚Üí `notifyExit()` + `std.process.exit(128 +% sig)` (`+%` is wrapping addition; `128+N` is the shell convention for "killed by signal N" so `$?` matches a normal process).
- **`SignalHandler` struct with function pointers** (`posix.zig:109-123`): bridges a C signal handler (plain function, C ABI, no closure state) to Zig struct methods ‚Äî stash state in a global (`g_signal_handler`), look it up in the handler, call through pointers.
- **`callconv(.c)`** (`posix_signals.zig:28`): Zig functions default to Zig calling convention (compiler-chosen register/stack layout). The kernel calls signal handlers via the **C ABI** (args on stack per platform C convention). `callconv(.c)` tells Zig to emit with C calling convention so the kernel can call it. Without it, stack corruption on call. Same reason `extern "c" fn` exists for FFI; `callconv(.c)` is the function-pointer version.
- **Taking `&` of a `pub const`** (`SIGNAL_SHUTDOWN` in `handleShutdown`): `&` forces Zig to materialize the const in rodata (read-only data section) with a stable address for the program's lifetime. `@ptrCast` reinterprets `*const u8` ‚Üí `[*]const u8`. Read-only memory is safe to *read* from a handler; const-ness guarantees no concurrent write; `write` is on the safe list. Fully async-signal-safe. (The const isn't inlined-away because `&` forces materialization.)
- **`g_signal_handler` write-once invariant** (`posix.zig:124`): `SignalHandler` is a multi-field struct ‚Äî writing it is **not atomic** (several store instructions). Safe today because `registerSignalHandlers` (line 146) does `g_signal_handler = handler` *before* `sigaction` enables delivery ‚Äî the kernel can't call `signalAction` until after the global is fully written. **Latent bug if anyone adds re-registration**: a write to `g_signal_handler` while signals are enabled would be a torn-struct race (some fields new, some old ‚Üí pointers to freed memory, garbage function addresses). Fix if ever needed: `sigprocmask(SIG_BLOCK, ...)` around the write, then restore. The conductor's self-pipe avoids this entirely ‚Äî handler does no struct dereference, just a one-byte write of a constant.
- **Windows: no signals at all.** Closest analogs: (a) **Console control events** ‚Äî `GenerateConsoleCtrlEvent` / `SetConsoleCtrlHandler`; Ctrl-C/Ctrl-Break generate `CTRL_C_EVENT`/`CTRL_BREAK_EVENT` delivered to a registered handler in a *separate thread* the kernel spawns (not an interruption of an existing thread ‚Üí no async-signal-safety constraint, but also no "inject into the middle of what I was doing"; normal synchronization usable in handler; self-pipe trick unnecessary). Only works for processes attached to a console. (b) **Termination** ‚Äî `TerminateProcess` (no cleanup, like SIGKILL), `WM_CLOSE` to GUI, `ExitProcess` from within. No graceful SIGTERM equivalent by default. (c) **No `kill(pid, sig)`** ‚Äî `OpenProcess` + `TerminateProcess` is always "kill" (no signal number). Interrupt to a worker ‚Üí named pipe message, Event object, or `GenerateConsoleCtrlEvent` if worker shares conductor's console.
- **Windows port implication**: entire `posix_signals.zig` becomes Windows-specific ‚Äî replace self-pipe with `SetConsoleCtrlHandler` (runs in own thread, can use normal I/O directly), replace `platform.kill(pid, SIG.INT)` in `handleNotification` with a Windows interrupt mechanism (pipe message to worker). The `worker_raw`/`\x03` client routing is *easier* on Windows ‚Äî `SetConsoleCtrlHandler` + writing to worker's stdin handle does the job, no signal-handler gymnastics.

### L8 ‚Äî the `std.Io` unification attempt (`client/eloop/{kqueue,linux}.zig`, `client/client.zig:19`, `conductor/main.zig:1944`)
- **Project uses Zig 0.16.0** ‚Äî has the new `std.Io` API (recent redesign of Zig's I/O abstraction). The boundary between `std.Io` and hand-rolled code *is* the Windows porting boundary.
- **The split ‚Äî std.Io creates, raw syscalls operate**: `Io.net.UnixAddress.init(path).connect(io)` does socket+connect under the hood, returns `Io.net.Stream` (cross-platform). But then the event loop cracks it open: `var server_fd: posix.fd_t = server.socket.handle;` (`conductor/eloop/kqueue.zig:112`) to register the raw fd with kqueue. Central pattern: **`std.Io` is used to *create* resources (sockets, files, pipes). Raw syscalls *operate* on them (multiplexing, signal-safe writes, terminal raw mode).**
- **What `std.Io` covers (the "easy" ~60%)**: network connect/listen/accept (`Io.net.UnixAddress`/`Io.net.IpAddress` ‚Äî likely cross-platform); filesystem (`Io.Dir.openDirAbsolute`/`deleteFileAbsolute`/`createDirAbsolute`/`createFileAbsolute`/`readFile`/`openFileAbsolute`/`createDirPath`/`cwd()` ‚Äî most portable); time (`Io.Clock.now(.awake, io)`, `.toSeconds()`, `.nanoseconds`); sleep (`Io.sleep(io, Io.Duration.fromMilliseconds(100), .awake)`); pipes (`Io.Threaded.pipe2`). These are operations where semantics are uniform across OSes; `std.Io` papers over the kernel-interface differences.
- **What's hand-rolled (the "hard" ~40%)**: event loops (kqueue/epoll/io_uring/IOCP have inverted semantics ‚Äî readiness vs completion, no clean abstraction); signal handling (signals don't exist on Windows; self-pipe is POSIX-only); raw socket I/O (`rawSocket`/`rawConnect`/`rawClose`/`rawIoctl`/`write` ‚Äî needed for async-signal-safety and because POSIX `write()` works on sockets but Windows doesn't); socket read/write/close (`platform.socketWrite`/`socketRead`/`close` ‚Äî POSIX unifies file/socket fds, Windows doesn't); terminal raw mode (`posix.tcgetattr`/`tcsetattr`/`termios` ‚Äî no Windows equivalent, uses `SetConsoleMode`); process groups (`.pgid = 0` ‚Äî POSIX-only, already gated `null` on Windows); process stats (`proc_pid_rusage` macOS / `/proc` Linux ‚Äî Windows uses `GetProcessMemoryInfo`).
- **The `Io` instance ‚Äî how it's created**: client (`client.zig:19`) `const io: Io = Io.Threaded.global_single_threaded.io();` ‚Äî compile-time-global, single-threaded, direct-syscall backend, no thread pool. Conductor (`main.zig:1944`) `pub fn main(init: std.process.Init) !void { const io = init.io; }` ‚Äî new Zig 0.16 `main` signature: runtime hands you initialized I/O context, allocator, environment map. The `init` pattern lets the runtime choose the backend; the client hardcodes because it *knows* it's single-threaded and wants zero overhead.
- **`Io` IS a vtable (runtime polymorphism, opt-in)**: an `Io` value is a fat struct ‚Äî context pointer + struct of function pointers. Calls through `io` are indirect calls (load fn ptr, call), not monomorphized static dispatch. Cost: one indirection per I/O call, dwarfed by the syscall cost. `Io.Threaded.global_single_threaded.io()` returns an `Io` whose vtable points at direct-syscall impl. Because it's a vtable, `std.Io`'s Windows backend is *pluggable at runtime without touching callers* ‚Äî if `Io.net.UnixAddress.connect()` works on Windows, it's because someone wrote the Windows fn-pointer table behind it. Question for Tier 1: **is that table complete in 0.16?** Empirical ‚Äî find out by cross-compiling and seeing what links.
- **Why event loops can't use `std.Io`**: compare the two client loops. **kqueue** (`client/eloop/kqueue.zig:39`): `makeKevent(@intCast(stdout_fd), c.EVFILT.READ, c.EV.ADD, ...)` ‚Äî "tell me when stdout_fd is *readable*" (readiness); you register interest, then `read()` yourself when kqueue says ready; the event gives you "readable, N bytes available" (`ev.data`), you still call `read()`. **io_uring** (`client/eloop/linux.zig:41`): `ring.read(user_data, fd, .{ .buffer = &buf }, 0)` ‚Äî "do a `read` on fd into this buffer, tell me when it's *done*" (completion); you submit the read *now*, kernel completes *later*; the completion gives you buffer + length (`cqe.res`). Different units of work (readiness notification vs completion result), different handle types (`fd_t` vs `HANDLE`), different registration lifecycles (register interest once vs submit SQE per op). **IOCP is completion-model like io_uring** ‚Äî so the io_uring loop is *closer* to what a Windows IOCP loop would look like than the kqueue loop is. But still different API (`GetQueuedCompletionStatus` vs `submit_and_wait`), handle types, registration. You can't write one loop body across all three without leaking the abstraction.
- **Windows port splits into two tiers**: **Tier 1 ‚Äî "should mostly work" (std.Io)**: filesystem ops (likely work or minor adjustments); time; sleep; high-level network connect/listen/accept (works *if* `std.Io`'s Windows backend is complete in 0.16 ‚Äî the question mark); pipe creation (Windows has anonymous pipes, but inheritance semantics for children differ). **Tier 2 ‚Äî "must be written from scratch"**: `client/eloop/windows.zig` (IOCP, ~150 lines, closest analog is the io_uring loop); `conductor/eloop/windows.zig` (IOCP + signal-equivalent, ~400 lines, biggest single piece); `platform/windows.zig` for both client+conductor (Winsock `WSASocket`/`connect`/`closesocket`, `GetStdHandle`, `defaultRuntimeDir` ‚Üí `%LOCALAPPDATA%\julia-daemon` or similar, `GetProcessMemoryInfo`, `SetConsoleMode`, `SetConsoleCtrlHandler`); the `kill(pid, SIG.INT)` ‚Üí Windows interrupt mechanism; named-pipe vs `AF_UNIX` decision. **Rough ratio: 60% uses `std.Io` (may need little change); 40% is hand-rolled (needs full Windows impls). But the 40% is the *hard* 40% ‚Äî all the subtle OS-internals.**

## Repository orientation map

A conceptual map for navigating the codebase. Organized by *layer within each component*, because that's how the platform abstraction cascades.

### Three components (README `Architecture`, line 199)
```
Client (Zig, client/)  ‚îÄ‚îÄ‚Üí  Conductor (Zig, conductor/)  ‚îÄ‚îÄ‚Üí  Worker Pool (Julia, worker/)
       ‚îÇ                          ‚îÇ                                   ‚îÇ
       ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ stdio ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
```

- **Client** (`juliaclient`): invoked by users. Connects to conductor, gets assigned a worker, connects to worker's 4 stdio sockets, proxies stdin/out/err + signals. Short-lived (one invocation = one client run).
- **Conductor** (`julia-conductor`): long-running daemon. Accepts clients, manages worker pool (spawn/evict/reap), routes clients to workers, pings workers for health, handles `--status`/`--restart`. The "brain".
- **Worker** (Julia, `worker/` + `DaemonWorker` module): a Julia process serving one project. Holds warm state. Created by conductor via connect-back (L5). Not Zig ‚Äî out of scope for the port.

### Key navigation tips
- **To understand a request's lifecycle**: start at `client.zig:main` ‚Üí `connectToConductor` ‚Üí `connectToWorker` ‚Üí `eloop.run`. Then conductor side: `main.zig:run` ‚Üí `eventLoopImpl.run` ‚Üí `handleAccept` ‚Üí `handleConnectionFd` ‚Üí `handleClient` ‚Üí `assignClientToWorker` ‚Üí `selectWorker` ‚Üí `Worker.runClient` ‚Üí `ClientStreams.setup` (the 4-listener dance).
- **To understand the platform abstraction**: read `platform/main.zig` first (the router), then one `impl` (`bsd.zig` is smallest), then `posix.zig` (shared). L1/L2 in `LESSONS.md`.
- **To understand the event loop**: read `client/eloop/kqueue.zig` (smallest, clearest) then `conductor/eloop/kqueue.zig` (adds timers/signals/pressure). L4/L7.
- **To plan the Windows port**: the `platform/main.zig` files (both client+conductor) are the index ‚Äî every `pub const`/`pub fn` there is a symbol `windows.zig` must export. L2's "two-tier" pattern tells you which come for free (none, on Windows) and which you must implement (all of them). Tier 1/Tier 2 split in L8 tells you what `std.Io` might cover.
